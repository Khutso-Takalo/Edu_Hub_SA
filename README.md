# React + TypeScript + Vite

This template provides a minimal setup to get React working in Vite with HMR and some ESLint rules.

## Security Configuration

Create a local environment file before running the app:

- Copy `.env.example` to `.env.local`.
- Set `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.
- Keep `VITE_ENABLE_CHAT_LEARNING_SYNC=false` unless your edge function is deployed and reviewed.
- Optional discovery settings:
  - `VITE_GOOGLE_CUSTOM_SEARCH_API_KEY`
  - `VITE_GOOGLE_CUSTOM_SEARCH_ENGINE_ID`
  - `VITE_GOOGLE_CUSTOM_SEARCH_BASE_URL`

### Chat learning privacy safeguards

- Global learning sync is opt-in via `VITE_ENABLE_CHAT_LEARNING_SYNC=true`.
- Synced events exclude raw user messages and token strings.
- Sync skips prompts that appear to contain sensitive patterns (email/phone/ID-like numbers).
- Local adaptive learning remains active even when global sync is disabled.

## Authentication Hardening

- Email inputs are normalized (`trim` + lowercase) before auth requests.
- Sign-up requires password policy: minimum 8 characters with letters and numbers.
- Sign-up includes password confirmation matching.
- Sign-in applies lightweight local throttling after repeated failures (short lockout window).
- Reset password responses use non-enumerating messaging (`If this email is registered...`).
- Reset and verification links are configured with `window.location.origin` redirect targets.

### MFA-ready scaffold

- Sign-in now supports a step-up path when MFA is required.
- Auth modal switches to a TOTP verification step (6-digit code) when the backend requests `aal2`.
- Logged-in users can manage MFA from **Security Settings** in the account dropdown menu.
- `useAuth` now exposes MFA helper methods for future profile/security settings pages:
  - `refreshMfaFactors`
  - `enrollMfaTotp`
  - `verifyMfaEnrollment`
  - `unenrollMfaFactor`

## Phase E: Data Freshness Workflow

The bursary dataset is local-first and can be refreshed through either scripts or the in-app admin panel.

### Scraper commands

- `npm run scrape:sources` generates a registry-driven source catalog snapshot at `src/data/seed/source-catalog.json` and a run report at `src/data/seed/source-catalog.report.json`.
- `npm run scrape:bursaries` generates a fresh scrape snapshot at `data/seed/bursaries.scraped.json` and a run report at `data/seed/bursaries.scrape-report.json`.
- `npm run scrape:bursaries:merge` merges scraped rows into `src/data/seed/bursaries.json` (the seed consumed by the app).
- `npm run data:check-links` validates bursary links and applies freshness penalties for broken links. Output report: `data/seed/bursaries.link-health-report.json`.
- `npm run data:reconcile` detects cross-source field conflicts and marks conflicting records for review. Output report: `data/seed/bursaries.reconciliation-report.json`.
- `npm run data:qa` runs both checks in sequence.

### Admin panel operations

In the **Admin & Data Lab** view you can:

- Run live source scrape (browser-safe keyword extraction + fallback rows).
- Review the registry-driven source catalog generated by the weekly Node scraper workflow.
- Export current bursary records from IndexedDB.
- Import bursary JSON for local ingestion.
- Review and resolve community outdated-data flags.

The panel updates metadata (`lastSeededAt`, `seedSummary`) for auditability and freshness badges.

## Phase I: Dynamic UI Experiment Analytics

Lightweight local analytics now capture dynamic UI experiment outcomes without external telemetry.

### Tracked metrics

- Hero search submit rate by mode (`new`, `curious`, `returning`).
- Suggestion-chip click-through rate (CTR).
- Urgency-ranked search results CTR.

### Where metrics are visible

- **Admin & Data Lab** -> **System Health Checklist** now shows:
  - suggestion CTR,
  - urgency CTR,
  - hero submit rate by mode.
  - variant split for hero submit, suggestion CTR, and urgency CTR.
  - analytics window filter (`7d`, `30d`, `all`).
  - trend sparklines for hero submit rate, suggestion CTR, and urgency CTR.
- Exported and copied health reports include a `uiExperiments` block.

### Storage and reset notes

- Analytics events are stored in browser `localStorage` under `eduhub:ui-analytics:v1`.
- Search history remains in `eduhub:search-history:v1` and powers adaptive suggestions.
- Admin diagnostics include a **Reset UI analytics** action that clears `eduhub:ui-analytics:v1` for fresh experiment cycles.
